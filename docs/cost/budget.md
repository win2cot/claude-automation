# API コスト管理

## 1. 想定 API 消費経路

| 経路 | 主体 | 想定頻度 |
|---|---|---|
| Cowork(対話) | 人が起動 | 不定期。議論・設計時に使用 |
| Claude Code(ローカル) | 人 / Cowork が起動 | 不定期。リポジトリ初期化や横断作業時 |
| GHA 実装 Claude | Issue/PR ラベルで自動起動 | PR 1 本あたり数回(初回 + 修正 n 回) |
| GHA レビュワ Claude | PR イベントで自動起動 | PR 1 本あたり数回(初回 + 再レビュー n 回) |

## 2. 月間予算(暫定)

- 全経路合計の月間上限: **$XX**(暫定。実運用 1 ヶ月で実績を計測してから設定)
- 各経路の内訳目安(暫定):
  - Cowork: 30%
  - Claude Code: 20%
  - GHA(実装 + レビュワ): 50%

## 3. アラート閾値(暫定)

| 段階 | 累計消費率 | アクション |
|---|---|---|
| 通知 | 50% | 自動通知 |
| 注意 | 80% | Cowork が朝の artifact で警告 |
| 警戒 | 95% | GHA workflow を一時無効化 |
| 停止 | 100% | 全自動 workflow 停止、手動運用に切替 |

## 4. 計測方法

- Anthropic API ダッシュボードで月間消費を確認
- GHA workflow run ログから 1 PR あたりのトークン消費を集計(別途 dashboard 構築は次フェーズ)
- Claude Code はローカルセッションログを参照

## 5. コスト抑制のテクニック

- **ラリー上限を厳格に**: 4 往復 → 3 往復に下げると消費が大きく減る
- **同一指摘検知を早く**: 2 回 → 1 回で `needs-human-decision`(精度とトレードオフ)
- **Issue 粒度を細かく**: 大 Issue ほどコンテキスト読み込みが膨らむ
- **規約・ADR を簡潔に**: プロンプトに含むため、冗長な記述はコスト直撃
- **不要な workflow を Required に入れない**: 検証時の試行錯誤コストが減る

## 6. 残課題

- 月間予算の具体値設定(運用 1 ヶ月後)
- 計測 dashboard の構築
- アラート自動通知の実装方式(GitHub Issue 集約 + `needs-human-decision` + @win2cot メンションに統一、詳細は ADR-0002 §2.6 通知ポリシー + SM-16 / SM-17 audit GHA で扱う)
